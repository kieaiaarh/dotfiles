# times-memo

## Overview

ユーザーの「思いついたこと」「ちょっとした予定」「リマインダー」を Slack の `#times_aihara` に整形済みで投稿するスキル。時刻が含まれる予定にはデフォルトで 30 分前のリマインダーを Slack 上に予約する。

**設計根拠:** `docs/superpowers/specs/2026-04-30-times-memo-design.md`

## When to Activate

以下のいずれかに該当する場合に**のみ**起動する。

- ユーザー発話が「メモ」で始まる（例: 「メモ 明日10時 田中さん」「メモ ブログのネタ：…」）
- 「〜をメモして」「〜をメモしておいて」「times にメモ」など、Slack 投稿依頼として明確
- スラッシュコマンド `/times-memo <内容>` で直接呼ばれた場合

### 起動しないケース（誤発火防止）

「メモ」という語が含まれていても、以下は起動対象外。判断に迷ったら **投稿前にユーザーに確認**を取る。

- 過去メモへの参照質問（例: 「昨日メモしてた件どうなった？」「メモした URL ある？」）
- コードや文書中の "memo" / "note" 語の編集依頼
- 通常の作業依頼（バグ修正、コードレビュー、説明、リファクタ等）
- 一般的な相談・雑談（「メモを取る習慣ってどう思う？」等）

## Categorization

メモの種別を以下のいずれかに分類する。

| 種別 | 判定条件 | 見出し |
|------|---------|--------|
| アイデア | 時刻語・期日語を含まない自由記述 | `💡 アイデアメモ` |
| 予定 | 「今日／明日／来週」等の日付語 + 時刻（`HH時` / `HH:MM` 等） | `📅 予定メモ` |
| リマインダー（時刻あり） | 上記「予定」に加え、「リマインドして」「忘れないように」等の語 | `⏰` 扱い（30 分前通知あり、見出しは `📅 予定メモ` のまま） |
| リマインダー（時刻なし） | 期日のみ／期日もなく「忘れないように」のみ | `⏰ リマインダー`（通知予約なし） |

判定が曖昧な場合は推定で進める（種別が違っても実害は小さい・速さ優先）。違っていればユーザーが Slack 側で編集すればよい。

## Time Parsing

予定／リマインダー判定時は、自然言語の時刻表現を **絶対日時（システムローカルタイム、JST 想定）** に変換する。

- 基準時刻: 会話コンテキストの `currentDate` とシステム時刻
- 「明日10時」「来週月曜 14:00」「30 分後」「月末まで」などを解釈
- 解釈に曖昧さがある場合は確認質問を出す
  - 例: 「メモ 10時 田中さん打ち合わせ」→ 「明日(5/1)の 10:00 で合ってますか？それとも今日の 22:00 ？」
- 12 時制（AM/PM 不明）も曖昧として扱う

絶対日時は Unix timestamp（秒）に変換して `slack_schedule_message` の `post_at` に渡す。

## Reminder Scheduling

予定／リマインダー（時刻あり）に該当するメモのみ Slack スケジュール送信を予約する。

- **デフォルトのリマインド時刻**: イベント時刻の **30 分前**
- **ユーザー明示時はそれを優先**: 「1 時間前にリマインド」「45 分前」「直前」等
- **Slack 制約への対応**:
  - `slack_schedule_message` の `post_at` は **現在 + 2 分以上** 必要
  - 計算結果がそれを下回る場合は「現在 + 2 分」に丸める。その旨をユーザーに通知
  - イベント時刻自体が過去になっている場合は **予約をスキップ**（即時投稿のみ）。ユーザーに通知
- **120 日より先**: スキップ（予約せず即時投稿のみ）

## Channel Resolution

`times_aihara` の channel_id を毎回解決する。

1. `mcp__claude_ai_Slack__slack_search_channels` を `query="times_aihara"` で呼ぶ
2. 返却の中から `name == "times_aihara"` を選択し `channel_id` を取得
3. 見つからない場合はユーザーに通知して中断（投稿しない）

## Post Format

### 共通ルール

- 1 行目: 絵文字 + 種別ラベル
- 2 行目以降: 本文（ユーザー発話を整形）
- 引用記号 `>` は使わない（モバイル可読性）
- タイムスタンプは入れない（Slack 側に投稿時刻が出るため冗長）
- `@user` メンションは入れない
- 予定メモのみ 3 行目にリマインド予約時刻を併記

### 例 1: アイデアメモ

```
💡 アイデアメモ
ブログのネタ: AIエージェントの設計について
```

### 例 2: 予定メモ（リマインダー予約あり）

```
📅 予定メモ
明日10時 田中さんと打ち合わせ
⏰ 2026-05-01 09:30 にリマインドします
```

### 例 3: リマインダー本体（30 分前に届く方）

```
⏰ もうすぐ: 田中さんと打ち合わせ（10:00〜）
```

### 例 4: リマインダーメモ（時刻なし）

```
⏰ リマインダー
請求書の処理を月末までに
```

## Workflow

1. **発動条件チェック** — When to Activate に従い起動可否を判断。曖昧なら確認
2. **種別判定** — Categorization 表に従う
3. **時刻解釈** — 予定／リマインダー時のみ。曖昧なら確認
4. **リマインダー時刻決定** — Reminder Scheduling のルールで `post_at` を決定（または予約スキップ）
5. **チャンネル ID 解決** — Channel Resolution
6. **投稿実行**
   - `mcp__claude_ai_Slack__slack_send_message` で即時投稿（Post Format に従う）
   - リマインダー予約が必要な場合: `mcp__claude_ai_Slack__slack_schedule_message` を呼ぶ
7. **結果報告** — ユーザーに簡潔に報告（例: 「投稿しました。リマインドは 2026-05-01 09:30 に届きます」）

## Confirmation Rules

| ケース | 例 | 挙動 |
|--------|-----|------|
| 発動判定が曖昧 | 「今日打ち合わせメモしてた件どうなった？」 | 「`#times_aihara` にメモを投稿しますか？」と確認 |
| 種別判定が曖昧 | 「メモ プロジェクトX 来週から始める」 | 推定で進める（確認なし） |
| 時刻解釈が曖昧 | 「メモ 10時 田中さん打ち合わせ」 | 「明日(5/1)の 10:00 で合ってますか？」と確認 |
| 過去時刻 | 「メモ 今日9時 ○○」（既に午後） | 即時投稿のみ、予約スキップ、結果で通知 |
| リマインド時刻明示 | 「メモ 明日10時 ○○、1時間前にリマインド」 | デフォルト 30 分前を上書きし 9:00 に予約 |
| 2 分制約に抵触 | 「メモ 3 分後に席を立つ」 | 「現在 + 2 分」に丸めて予約、ユーザーに通知 |
| 120 日超 | 「メモ 半年後の○○」 | 予約スキップ、即時投稿のみ、ユーザーに通知 |
| `times_aihara` が見つからない | — | 投稿せず中断、ユーザーに通知 |

## Out of Scope

このスキルでは扱わない（明示的に対応しない）。

- 他チャンネルへの投稿（`#times_aihara` 固定）
- 画像・添付ファイル付き投稿
- スレッド返信
- 過去メモの検索・編集（投稿後の編集は Slack 側でユーザーが実施）
- リマインダー以外の繰り返しスケジュール（毎週／毎月）
- 投稿前のドラフトプレビュー（速さ優先）
