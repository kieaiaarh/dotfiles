## 制約
* 必ず最新のファクトベースで考え、回答をすること

## 実装前の確認原則
* 実装に入る前に、設計方針・変更差分をユーザーに提示してから進む
* 提案のみを求められている場合はコードを書かず、文章で提示する
* スコープ外の改善点に気づいた場合は実装せず、口頭で提案にとどめる

## 破壊的操作の原則
* ファイル削除・ブランチ削除・DB変更・本番環境への操作は、実行前に必ずユーザーに確認を取る
* 「まとめて削除」「全件更新」「reset --hard」等は承認なしに実行しない
* 確認なしで実行してよいのは、ローカルかつ可逆性のある操作のみ

## sandbox 制約で失敗した場合の報告フォーマット
1. 実行しようとした Make ターゲット / コマンド
2. sandbox 制約で失敗したこと
3. ユーザーがローカルで実行すべきコマンド

## 完了前の確認
* 新規スクリプト・機能の実装完了前、PR・コミット前に必ず `superpowers:verification-before-completion` スキルを呼ぶ
* シェルスクリプト（.sh）を作成・編集した場合、プロジェクトの `.claude/rules/shell.md` のセルフチェックリストを確認してから PR を出す

## 回答スタイル
* 回答は簡潔に。説明が長くなる場合は最初に結論を述べる
* 「〜しました」の末尾要約は不要。差分を見ればわかる

## CLIコマンドの提示ルール
* CLIコマンドのオプションを提示する前に、必ず `--help` を実行して実際に使えるオプションを確認してから提示する
* 知識から推測してオプションを提示しない

## PR / Issue 記述ルール
* 平易な日本語で書く。専門用語の羅列を避け、利用者視点を優先する
* 各リポジトリの `.github/PULL_REQUEST_TEMPLATE.md` / `.github/ISSUE_TEMPLATE/` の構造に従うこと
* `gh pr create --body` / `gh issue create --body` で本文を直接渡す場合も、テンプレ構造を踏襲する
* PR の構成順序：「概要（なぜ）→ 修正内容（何が変わるか・利用者視点）→ やらないこと → 影響範囲 → 動作確認 → 備考」
* 「概要」「修正内容」には**実装詳細を書かない**。実装詳細は「動作確認」セクションに集約する
* 「概要」は「実装内容のサマリ」ではなく**「なぜ必要か」**を書く
* 既存セクション名（概要 / 修正内容 / やらないこと / 影響範囲 / 備考）はリポ間で揃える
