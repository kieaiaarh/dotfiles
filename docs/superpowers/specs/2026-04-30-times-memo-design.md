# times-memo スキル 設計書

- 起票日: 2026-04-30
- 対象: ユーザー個人（gouharakei）の Slack `#times_aihara` 投稿用 Claude Code スキル
- 配置先: `~/work/buzzkuri/dotfiles/ai/claude/skills/times-memo/SKILL.md`（実体）/ `~/.claude/skills/times-memo/`（シンボリックリンク）

## 1. なぜ作るか（目的）

ユーザーは Claude Code 上で音声・テキスト入力を主に行う運用で、思いついたこと・ちょっとした予定・リマインダーを Slack の `#times_aihara` に「気軽に・見やすい形で」流したい。現状はユーザー自身が Slack に転記するか、毎回 Claude に詳細な投稿指示を出す必要があり、メモのハードルが高い。

## 2. 達成したい状態

- ユーザーが「メモ 〇〇」と一言伝えれば、Claude が自動で `#times_aihara` に整形済みのメモを投稿する
- 時刻が含まれる場合は 30 分前に Slack リマインダーが自動で届く
- 通常の作業依頼（バグ修正・コード説明など）では誤発火しない
- 判断に迷ったケースでは、投稿前にユーザーに確認を取る

## 3. スキルメタデータ（フロントマター）

公式 [Claude Code Skills](https://code.claude.com/docs/en/skills) 仕様準拠。

```yaml
---
name: times-memo
description: ユーザーの思いつき・予定・リマインダーを Slack の #times_aihara に見やすく投稿する。発動条件は厳密で、ユーザー発話が「メモ」で始まる／「〜をメモ」「メモして」など明示的なメモ依頼の場合のみ。通常の作業依頼や質問には発動しない。
when_to_use: ユーザーが「メモ 明日10時に田中さんと打ち合わせ」「ブログのネタをメモして」のように、思いついたこと・予定・リマインダーを Slack の times_aihara に流したいと明示した時のみ。判断に迷ったら投稿前にユーザーに確認する。
argument-hint: [メモ内容]
allowed-tools:
  - mcp__claude_ai_Slack__slack_search_channels
  - mcp__claude_ai_Slack__slack_send_message
  - mcp__claude_ai_Slack__slack_schedule_message
---
```

設計上の判断:

- `disable-model-invocation` は明示しない（=デフォルト `false`）。自然言語トリガー「メモ」で発動できるようにする
- 公式は副作用ありスキルでの自動起動 OFF を推奨しているが、本スキルはユーザー発話の「メモ」自体が明示的なオプトイン信号となるため、description で発動条件を厳密化することで誤発火を抑える
- `allowed-tools` で Slack MCP ツールを事前承認し、都度の権限プロンプトを省略

## 4. メモ種別と判定ロジック

| 種別 | 判定条件 | 見出し |
|------|----------|--------|
| アイデア | 時刻語・期日語を含まない自由記述 | 💡 アイデアメモ |
| 予定 | 「今日／明日／来週」等の日付語 + 時刻（`HH時`, `HH:MM` 等） | 📅 予定メモ |
| リマインダー（時刻あり） | 上記「予定」に加え、「リマインドして」「忘れないように」等の明示語がある | ⏰ リマインダー扱い（30分前通知あり） |
| リマインダー（時刻なし） | 期日のみ・期日もなく「忘れないように」のみ | ⏰ リマインダー（通知予約なし） |

判断が曖昧なケースは Claude 推定で進める（種別が違っても実害が小さい・速さ優先）。違っていればユーザーが Slack 側で編集する。

## 5. 投稿フォーマット

### 5.1 アイデアメモ

```
💡 アイデアメモ
ブログのネタ: AIエージェントの設計について
```

### 5.2 予定メモ（リマインダー予約あり）

```
📅 予定メモ
明日10時 田中さんと打ち合わせ
⏰ 2026-05-01 09:30 にリマインドします
```

### 5.3 リマインダー本体（30分前に届く方）

```
⏰ もうすぐ: 田中さんと打ち合わせ（10:00〜）
```

### 5.4 リマインダーメモ（時刻なし）

```
⏰ リマインダー
請求書の処理を月末までに
```

### 5.5 共通ルール

- 引用記号（`>`）は使わない（モバイルで見づらい）
- タイムスタンプは入れない（Slack 側に投稿時刻が表示されるため冗長）
- `@user` メンションは入れない
- 1 行目: 絵文字 + 種別ラベル
- 2 行目以降: 本文
- 予定メモのみ 3 行目にリマインド予約時刻を併記

## 6. 処理フロー

```
1. 発動条件チェック
   - 入力に「メモ」トリガーがあり、メモ依頼として明確か確認
   - 不明確な場合は「times_aihara にメモしますか？」とユーザーに確認

2. メモ種別判定（5章のロジックに基づく）

3. 時刻解釈（予定／リマインダー時のみ）
   - 「明日10時」「来週月曜 14:00」等を絶対日時に変換
   - 基準時刻はシステム時刻（currentDate 含む）
   - 解釈に曖昧さがある場合は確認（例: 「明日(5/1)の 10:00 で合ってますか？」）

4. リマインダー時刻決定
   - デフォルト: イベント時刻の 30 分前
   - ユーザーが「○分前」「○時間前」を明示した場合はそれを優先
   - 過去時刻になる場合は予約スキップ + ユーザー通知

5. チャンネルID解決
   - slack_search_channels で "times_aihara" を検索しID取得

6. 投稿実行
   - slack_send_message で即時投稿
   - リマインダー予約が必要な場合: slack_schedule_message で予約

7. 結果報告
   - 「投稿しました（リマインドは○○に届きます）」とユーザーに簡潔に報告
```

## 7. 確認フローのケース別整理

| ケース | 例 | 挙動 |
|--------|-----|------|
| 発動判定が曖昧 | 「今日打ち合わせメモしてた件どうなった？」 | 「times_aihara にメモを投稿しますか？」と確認 |
| 種別判定が曖昧 | 「メモ プロジェクトX 来週から始める」 | Claude推定で進める（確認なし） |
| 時刻解釈が曖昧 | 「メモ 10時 田中さん打ち合わせ」 | 「明日(5/1)の10:00 で合ってますか？」と確認 |
| 過去時刻 | 「メモ 今日9時 ○○」（既に午後） | 即時投稿のみ実施、リマインド予約はスキップ、結果で通知 |
| リマインド時刻明示 | 「メモ 明日10時 ○○、1時間前にリマインド」 | デフォルト30分前を上書きし、9:00 に予約 |

## 8. やらないこと（スコープ外）

- 他チャンネルへの投稿（`#times_aihara` 固定）
- 画像・添付ファイル付き投稿
- スレッド返信
- 過去メモの検索・編集（投稿後の編集はユーザーが Slack 側で実施）
- リマインダー以外の繰り返しスケジュール（毎週/毎月など）
- 投稿前のドラフトプレビュー（速さ優先のため都度確認はしない）
