# コンプライアンス検知ロジック刷新計画

## 目標

- 9カテゴリ → 3カテゴリ（パワハラ/セクハラ/その他）への再定義
- 大量辞書生成によるFalse Negative削減
- AIコストの大幅削減（月300万円 → 数万円レベル）
- Slack/Teams特有の表現に対応

## 実装ステップ

### STEP 1: カテゴリ再定義とDBスキーマ変更

**1-1. domain_key enumの変更**

- `app/models/compliances/detection_rule.rb`のenumを変更
- 既存: `harassment: 0, information_leakage: 1, ...` (9カテゴリ)
- 新規: `power_harassment: 0, sexual_harassment: 1, others: 2` (3カテゴリ)
- 既存データの移行マッピングを定義（harassment → power_harassment など）

**1-2. Ridgepoleスキーマ更新**

- `db/Schemafile`の`compliance_detection_rules`テーブルのコメントを更新
- 既存データの移行スクリプト作成

**1-3. パワハラ軸の定義（12軸）**

- `lib/tasks/compliances/power_harassment_axes.rake`を作成
- 12軸の定義をYAML/JSON形式で保存
  - 侮辱・人格否定
  - 脅迫・威圧
  - 暴言・罵倒
  - 見下し・軽視
  - 無能扱い・能力否定
  - 過度な叱責・公開処罰
  - 無視・排除・孤立
  - グループ圧・晒し
  - Slack/Teams特有の圧（草/w/絵文字圧/皮肉スタンプ）
  - マイクロアグレッション（婉曲的侮辱）
  - 懲罰的労働強要
  - 退職強要・排除強要

### STEP 2: 自社Slackデータの分析と活用

**2-1. Slackデータ分析スクリプト**

- `lib/tasks/compliances/analyze_slack_data.rake`を作成
- 半年分のSlackメッセージ（`compliance_slack_messages`テーブル）を分析
- 既存ルールでヒットしなかったメッセージを抽出
- パワハラ的な表現パターンを統計的に分析
- 実際に使われている表現をリスト化

**2-2. データ抽出と前処理**

- メッセージテキストの正規化
- 頻出パターンの抽出
- 文脈情報（チャンネル、スレッド）の活用

### STEP 3: Knowledge Basesを使った辞書生成基盤

**3-1. カテゴリ説明Markdownの作成**

- `docs/knowledge_bases/power_harassment.md`を作成（AIが協力して作成）
- 日本の職場コミュニケーション前提
- Slack/Teams両方の文化差異を明記
- 典型NG / グレー / Safe例を含める
- 「圧」「晒し」「非言語ハラスメント」の説明
- **最近の若者言葉やネットスラングも加味**（例: 「草」「w」「マジで」「やばい」「ウケる」など）
- 12軸の定義に基づいた詳細な説明を記載
- 自社Slackデータの分析結果も反映

**3-2. 辞書生成プロンプトの設計**

- `lib/tasks/compliances/generate_dictionary.rake`を作成
- AWS Bedrock Knowledge Bases API連携
- 軸ごとにLLMで表現生成（100〜300語/回）
- 中間JSON形式で返すようプロンプト指定
- 自社Slackデータの分析結果も参照

**3-3. 中間JSON形式の定義**

```json
{
  "domain_key": "power_harassment",
  "risk_type": "verbal",
  "axes": [
    {
      "axis_key": "insult",
      "axis_label": "侮辱",
      "policy_key": "PH-HAR-001",
      "patterns": [
        { "type": "word", "text": "バカ", "strength": "strong" },
        { "type": "phrase", "text": "お前バカだろ", "strength": "strong" },
        { "type": "regex", "text": "草+$", "strength": "weak" }
      ]
    }
  ]
}
```

### STEP 4: 中間JSON → DB投入パイプライン

**4-1. JSON投入Rakeタスク**

- `lib/tasks/compliances/import_detection_rules_from_json.rake`を作成
- 中間JSONファイルを読み込み
- `compliance_detection_rules`テーブルに変換して投入
- 変換ルール:
  - `domain_key` → enum値（power_harassment=0）
  - `rule_type` → word/phrase=keyword(0), regex=1
  - `pattern` → テキスト
  - `base_score` → strength（strong=95, medium=70, weak=40）
  - `metadata_json` → policy_key, axis_key, axis_label, sourceを含む

**4-2. 重複チェックとマージ**

- 既存ルールとの重複をチェック
- 同一パターンは更新、新規は追加

### STEP 5: ContentNormalizerの強化

**5-1. Slack/Teams差異の吸収**

- `app/services/compliances/detectors/content_normalizer.rb`を拡張
- 絵文字 → `<emoji>`に正規化
- メンション（`<@U12345>`）→ `<mention>`に正規化
- URL → `<url>`に正規化
- スタンプ（`:emoji:`）→ `<stamp>`に正規化
- 全角→半角、カナ統一は既存実装を維持

**5-2. テストの追加**

- `spec/services/compliances/detectors/content_normalizer_spec.rb`を拡張
- Slack/Teams特有のパターンのテストケース追加

### STEP 6: KeywordDetectorの改善

**6-1. 長いパターン優先処理**

- `app/services/compliances/detectors/keyword_detector.rb`を修正
- ルールをパターン長でソート（降順）
- より長いパターンを優先的にマッチング
- 誤爆軽減と確実なヒットを実現

**6-2. テストの追加**

- `spec/services/compliances/detectors/keyword_detector_spec.rb`を拡張
- 長いパターン優先のテストケース追加

### STEP 7: 動作確認と検証

**7-1. 既存ルールとの整合性確認**

- 既存の検知ルールが正常に動作することを確認
- 新規ルールの検知精度を検証

**7-2. パフォーマンステスト**

- 大量ルール投入時の検知速度を確認
- メモリ使用量の確認

- テストカバレッジの確保
