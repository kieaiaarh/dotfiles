# コンプライアンス違反検知システムの改修（4テーブル設計）

## 目的

- 高精度かつ低コストな検知: ルールベース検知をベースに、GPT-4o miniによるAI補完で検知漏れを防止
- 柔軟な運用と低誤検知: 組織単位でルールを柔軟に管理し、誤検知を抑制

## 検知戦略

1. **第一段階（ルールベース検知）**: 除外ルール→組織別NGルール→共通ルール（上書き適用済み）の優先順位で適用
2. **第二段階（AI補完検知）**: ルール非ヒット時のみGPT-4o miniで判定
3. **第三段階（ルール昇格フィードバック）**: AI検知をログに蓄積し、昇格候補として提示

## テーブル設計（4テーブル構成）

1. **`compliance_detection_rules`**（共通ルール）

- 全組織に適用される共通の検知ルール
- `synonyms_json`カラムで類義語を管理

2. **`compliance_organization_detection_rules`**（共通ルール上書き専用）

- 共通ルールに対するスコア上書きのみ
- `detection_rule_id`は必須（NULL不可）
- `override_enabled`: 共通ルールの有効/無効を上書き
- `override_score`: 共通ルールのスコアを上書き
- **独自ルール機能は削除**（NGルールは別テーブルに分離）

3. **`compliance_organization_ng_rules`**（組織独自NGルール）

- 組織固有のNG強化ルール
- `domain_key`, `risk_type`, `rule_type`, `pattern`, `base_score`必須
- `synonyms_json`で類義語を管理
- 昇格候補からの登録先

4. **`compliance_organization_exclusion_rules`**（組織独自除外ルール）

- 組織固有の除外ルール
- `pattern`, `reason`必須
- `synonyms_json`で類義語を管理

## 実装のポイント

1. **ルール適用優先度**: 除外ルール（1st）→ 組織別NGルール（2nd）→ 共通ルール（3rd、上書き適用済み）
2. **synonyms_json対応**: JSON配列をパースし、patternとsynonymsの両方をマッチング
3. **AI補完検知**: ルール非ヒット時のみ実行、YES/NO判定で違反検知
4. **昇格候補**: 可変閾値（デフォルト3回）でAI検知を昇格候補として扱う。昇格時は`compliance_organization_ng_rules`に登録
5. **GPT-4o mini**: Bedrock経由で使用、モデルIDは設定から読み込み
6. **共通ルール上書き**: `compliance_organization_detection_rules`は上書き専用（`detection_rule_id`必須）

## 実装ガイドライン

実装では以下の原則を遵守すること：

- **宣言的であること**: 手続き的ではなく、何をしたいかを明確に表現する
- **Readableであること**: コードの可読性を最優先に、意図が明確に伝わるように書く
- **単一責任の原則**: 各クラス・メソッドは単一の責務を持つ。100行超のクラスは責務過多の可能性を疑う
- **Ruby 3記法**: ワンライナーメソッド（`def method = expression`）やハッシュ省略記法（`{ key: }`）を適宜使用する
- **コメントよりも宣言的**: コメントで説明するよりも、メソッド名や変数名で意図を明確に表現する
